# Security Design

本ドキュメントは 369web-prototype-90k-salon（架空ヘアサロン「Salon Lumière Kawagoe」デモサイト）のセキュリティ設計方針を記載する。

> ⚠️ **重要警告**
> 本実装は **営業用デモサイト** であり、**本番運用のセキュリティ要件は満たしていない**。
> 本コードベースを実プロダクトに転用する場合、必ず本ドキュメント末尾の「本番化時の必須対応」を実施すること。

---

## 1. 認証方式

### 採用方式
- **localStorage ベースの簡易認証**
- ユーザー情報・セッション情報はすべてブラウザ localStorage に保存

### 採用理由
- 月額サブスク 3,000円/月の制約内で運用するため、サーバーサイドDB・認証基盤を使用しない
- 営業デモとして「動く会員ページ」を提示できれば十分

### 本番非推奨理由
- localStorage は同一オリジン内の全 JavaScript から読み取り可能（XSS 脆弱性に弱い）
- ブラウザを変えるとログインできない（端末間同期不可）
- パスワードリセット・メール認証等のフローが存在しない

---

## 2. パスワード保存

### 採用方式
- `btoa()` による簡易エンコード（Base64 変換のみ・ハッシュではない）

### 採用理由
- デモとして「ログインできる」挙動を再現するだけが目的
- サーバーサイドDB がないため、本格的なハッシュ化（bcrypt 等）は不要

### 本番非推奨理由
- Base64 は誰でも復号可能（パスワードを平文と同等にしか保護していない）
- ソルト・ストレッチング処理が存在しない

### 警告コメントの配置
- `lib/auth.ts`（Phase 5 で生成予定）の冒頭に必ず以下のコメントを記載すること:
```
  // ⚠️ DEMO ONLY: 本実装はデモ用です。
  // 本番では bcrypt 等の適切なパスワードハッシュ化、
  // サーバーサイド認証、HTTPS-only Cookie 等を必須としてください。
```

---

## 3. データ漏洩リスク

### localStorage の脆弱性
- XSS（クロスサイトスクリプティング）に対して脆弱
- ブラウザの拡張機能・別タブの同一オリジン JS からアクセス可能
- バックアップ・暗号化されない

### 本デモでの取り扱い
- 保存する情報はすべて **ダミーデータ**（実在の個人情報を含まない）
- デモアカウント以外で実在のメールアドレス・パスワードを入力しないよう、UI に注意書きを表示する（Phase 5 で実装）

### 本番非推奨理由
- 個人情報保護法・GDPR 等の規制対象となる情報は localStorage に保存してはならない

---

## 4. デモアカウント

### 認証情報
| 項目 | 値 |
|---|---|
| Email | `demo@example.com` |
| Password | `demo1234` |
| Display Name | デモ太郎 |

### 用途
- 営業先で即座にマイページの挙動を見せるため
- /login ページに「デモアカウントでログイン」ボタンを設置し、ワンクリックで認証可能にする

### セキュリティ上の許容
- 本アカウントの情報は **公開デモ用** であり、秘匿性を持たない
- 本デモサイト以外で同じパスワードを使ってはならない（汎用パスワードではない）

---

## 5. noindex 運用

### 設定箇所
- `app/layout.tsx` の `metadata.robots` で `index: false, follow: false` を設定

### 採用理由
- 本デモサイトが検索エンジンに掲載されると、369WEB の正式 LP と混同される
- 「Salon Lumière Kawagoe」が架空店舗であることが伝わらず、誤情報を拡散するリスクがある

### 検証方法
- ブラウザで本番URL を開き、View Source で以下を確認:
```html
  <meta name="robots" content="noindex, nofollow">
```

### 本番LP（369WEB.jp 等）への切り替え時
- 本番LP は逆に `index: true, follow: true` とすること
- robots.txt や sitemap.xml も検索エンジンに公開する設定に変更

---

## 6. 秘密情報の取り扱い

### 本プロジェクトでの方針
- **環境変数は使用しない**
- API キー・トークン・データベース接続文字列を一切持たない
- 外部 API（OpenAI / Stripe / LINE 等）は呼ばない（Phase 6 のチャットボットも完全クライアントサイド完結）

### 記載ルール
- ドキュメント・コード・コミットメッセージ・チャット履歴で秘密情報を扱う場合は必ず `[REDACTED]` と表記すること
- 例: `API_KEY=[REDACTED]`

### 本番化時
- Stripe Price ID 等を導入する場合は Vercel の Environment Variables で管理（コードに直書きしない）

---

## 7. CORS / CSP

### 本プロジェクトでの方針
- 外部 API を呼ばないため、CORS 設定不要
- next.config.js のデフォルト設定で運用

### 本番化時
- Stripe / LINE 等の外部 API を導入する場合、CSP（Content Security Policy）ヘッダで許可ドメインを明示すること

---

## 8. 監査ログ

### 本プロジェクトでの方針
- 監査ログは取得しない（デモ用のため）

### 本番化時
- ログイン試行・パスワード変更・データ削除等の重要操作は監査ログに記録すること

---

## 9. 本番化時の必須対応チェックリスト

本コードを本番プロダクトに転用する場合、以下を必ず実施すること。

- [ ] localStorage 認証 → Supabase / Firebase Auth / NextAuth 等のサーバーサイド認証に置き換え
- [ ] パスワード → bcrypt / Argon2 等でハッシュ化（ソルト・ストレッチング含む）
- [ ] HTTPS-only Secure Cookie でセッション管理
- [ ] CSRF トークン導入
- [ ] レート制限（ログイン試行回数制限）
- [ ] パスワードリセット機能（メール認証経由）
- [ ] 2要素認証（任意）
- [ ] 個人情報の暗号化保存
- [ ] 監査ログの実装
- [ ] CSP ヘッダの設定
- [ ] 環境変数で秘密情報を管理
- [ ] robots.txt / sitemap.xml の公開設定
- [ ] noindex 設定の解除（本番LPとして公開する場合）
- [ ] 利用規約・プライバシーポリシーの法務チェック
